# Brain 1 Spec: Movie Meaning

## Purpose

Brain 1 is the semantic retrieval brain.

It answers:

- what does a search query mean?
- which movies are semantically similar to a movie the user is viewing?
- which catalog items should enter the recommendation candidate pool before personalization?

For this repo, Brain 1 is the strongest phase-1 foundation because it already aligns with the current backend search implementation.

## Current repo reality

These examples intentionally match the current repo and MongoDB setup.

- collection: `embedded_movies`
- vector index name: `vector_index`
- embedding field: `plot_embedding`
- collection `_id`: `ObjectId`
- current backend config source: `backend/src/main/resources/application.yaml`
- current vector service anchor: `backend/src/main/java/com/hackathon/backend/services/VectorSearchService.java`

Important consequence:

- Brain 1 examples can query `embedded_movies` directly without type conversion because the search source collection already stores the embeddings on the movie documents.

## Why Brain 1 is mostly `$vectorSearch`

Aggregation is useful after retrieval.

But Brain 1's main job is not counting or grouping. Its main job is semantic nearest-neighbor retrieval.

That means the core pattern is:

1. create a query vector outside MongoDB
2. run `$vectorSearch` as the first stage
3. optionally filter and trim the result document shape
4. pass the candidate list to the backend ranker

## Near-runnable pipeline A: Search query to semantic candidates

### When to use it

- `GET /api/search/movies`
- homepage candidate generation from a recent search query
- any surface where free-text intent is the entry point

### Inputs

- `QUERY_VECTOR`: generated by the embedding service from user query text
- `LIMIT`: usually 10 to 40

### Pipeline

```javascript
db.embedded_movies.aggregate([
  {
    $vectorSearch: {
      index: "vector_index",
      path: "plot_embedding",
      queryVector: QUERY_VECTOR,
      numCandidates: 200,
      limit: 40,
      filter: {
        type: "movie"
      }
    }
  },
  {
    $addFields: {
      semanticScore: { $meta: "vectorSearchScore" }
    }
  },
  {
    $project: {
      plot_embedding: 0,
      plot_embedding_voyage_3_large: 0
    }
  },
  {
    $project: {
      _id: 1,
      title: 1,
      plot: 1,
      fullplot: 1,
      genres: 1,
      poster: 1,
      year: 1,
      imdb: 1,
      semanticScore: 1
    }
  }
])
```

### Stage-by-stage explanation

#### Stage 1: `$vectorSearch`

```javascript
{
  $vectorSearch: {
    index: "vector_index",
    path: "plot_embedding",
    queryVector: QUERY_VECTOR,
    numCandidates: 200,
    limit: 40,
    filter: {
      type: "movie"
    }
  }
}
```

Why this stage exists:

- this is the actual semantic retrieval step
- the query embedding is compared against `embedded_movies.plot_embedding`
- `type: "movie"` is a real field in the sample data and is safer than the earlier placeholder `availability.isActive`

Why this is accurate for MongoDB:

- `$vectorSearch` is allowed only as the first stage in the pipeline where it appears
- `index`, `path`, `queryVector`, `numCandidates`, and `limit` are valid `$vectorSearch` fields

Why `numCandidates` is 200 while `limit` is 40:

- this repo already defaults `default-num-candidates` to `200`
- asking for a larger candidate pool before final trimming usually improves ANN quality

Output of this stage:

- up to 40 candidate movie documents
- each document carries a vector similarity score exposed later through `$meta: "vectorSearchScore"`

#### Stage 2: `$addFields`

```javascript
{
  $addFields: {
    semanticScore: { $meta: "vectorSearchScore" }
  }
}
```

Why this stage exists:

- the current backend ranker needs the score as a normal field
- naming it `semanticScore` makes the later merge formula clearer than reusing `vectorSearchScore`

Output:

- same candidate documents plus `semanticScore`

#### Stage 3: first `$project`

```javascript
{
  $project: {
    plot_embedding: 0,
    plot_embedding_voyage_3_large: 0
  }
}
```

Why this stage exists:

- embeddings are large and should not be returned to the client or carried through later ranking if not needed
- this mirrors the current backend service behavior

Output:

- leaner candidate documents

#### Stage 4: second `$project`

```javascript
{
  $project: {
    _id: 1,
    title: 1,
    plot: 1,
    fullplot: 1,
    genres: 1,
    poster: 1,
    year: 1,
    imdb: 1,
    semanticScore: 1
  }
}
```

Why this stage exists:

- keep only fields the API and reranker actually need
- makes the result shape stable for downstream code

### Tradeoffs made

- using `type: "movie"` is a minimal safe filter, but it does not hide low-quality or incomplete records
- this pipeline returns semantically relevant items, not personalized items
- no dedupe by franchise, actor, or title family is applied here

### What we would improve later

- add metadata filters for language, region, or year if the product needs them
- introduce hybrid lexical + vector retrieval if query wording quality becomes a problem
- add business filtering such as availability or safe-content rules once the catalog carries those fields

## Near-runnable pipeline B: Current movie to similar movies

### When to use it

- detail page recommendations
- "more like this" rows

### Inputs

- `CURRENT_MOVIE_VECTOR`
- `CURRENT_MOVIE_ID`

### Pipeline

```javascript
db.embedded_movies.aggregate([
  {
    $vectorSearch: {
      index: "vector_index",
      path: "plot_embedding",
      queryVector: CURRENT_MOVIE_VECTOR,
      numCandidates: 200,
      limit: 30,
      filter: {
        type: "movie"
      }
    }
  },
  {
    $addFields: {
      semanticScore: { $meta: "vectorSearchScore" }
    }
  },
  {
    $match: {
      _id: { $ne: CURRENT_MOVIE_ID }
    }
  },
  {
    $project: {
      plot_embedding: 0,
      plot_embedding_voyage_3_large: 0
    }
  },
  {
    $project: {
      _id: 1,
      title: 1,
      genres: 1,
      poster: 1,
      year: 1,
      imdb: 1,
      semanticScore: 1
    }
  }
])
```

### Stage-by-stage explanation

#### `$vectorSearch`

- uses the current movie embedding as the query vector
- returns movies whose `plot_embedding` is nearest to the seed movie

#### `$addFields`

- exposes the similarity score as `semanticScore`

#### `$match`

- excludes the seed movie itself
- this must happen after `$vectorSearch` because `$vectorSearch` must remain the first stage

#### `$project` stages

- first remove heavy embedding fields
- then keep only response fields needed by the detail page

### Tradeoffs made

- excluding the seed movie after retrieval means one candidate slot is effectively wasted when the seed movie is returned by vector search
- that is acceptable in MVP because it keeps the query simple and valid

### What we would improve later

- slightly increase `limit` at retrieval time and trim after exclusion so the visible result count stays stable
- blend in user taste so similar movies become both semantically relevant and personalized

## Near-runnable pipeline C: Text fallback when embeddings or vector search fail

### When to use it

- embedding service timeout
- empty embedding output
- Atlas Vector Search outage or degraded mode

### Required prerequisite

This fallback is only useful if a text index exists on fields such as:

- `title`
- `plot`
- `fullplot`
- optionally `genres`

### Pipeline

```javascript
db.embedded_movies.aggregate([
  {
    $match: {
      $text: {
        $search: QUERY_TEXT
      },
      type: "movie"
    }
  },
  {
    $addFields: {
      textScore: { $meta: "textScore" }
    }
  },
  {
    $sort: {
      textScore: -1,
      "imdb.rating": -1
    }
  },
  {
    $limit: 40
  },
  {
    $project: {
      plot_embedding: 0,
      plot_embedding_voyage_3_large: 0
    }
  },
  {
    $project: {
      _id: 1,
      title: 1,
      plot: 1,
      genres: 1,
      poster: 1,
      imdb: 1,
      textScore: 1
    }
  }
])
```

### Why this is accurate

- `$text` is valid inside `$match`
- `$meta: "textScore"` can be exposed and sorted
- the current backend already falls back to text search in `MovieSearchService`

### Tradeoffs made

- lexical fallback is less robust for meaning-level search than vector retrieval
- if the text index is missing, this path will still fail and the service must fall back again to trending or top-rated items

## Required indexes

### Atlas Vector Search index

- collection: `embedded_movies`
- path: `plot_embedding`
- index name: `vector_index`

### Optional text index for fallback

Example:

```javascript
db.embedded_movies.createIndex({
  title: "text",
  plot: "text",
  fullplot: "text",
  genres: "text"
})
```

### Optional normal index

If `type: "movie"` remains a common filter outside vector search, a normal index can still help on non-vector fallback queries:

```javascript
db.embedded_movies.createIndex({ type: 1 })
```

## MVP limits

Brain 1 is strong, but it still has limits.

- it understands content similarity better than audience behavior
- it may overfavor obvious semantic matches
- it does not know what this specific user currently wants unless Brain 2 reranks the candidates

That is why Brain 1 should generate candidates first, then let Brain 2 personalize them.
